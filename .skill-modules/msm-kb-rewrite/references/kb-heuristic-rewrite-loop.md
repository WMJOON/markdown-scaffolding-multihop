# KB Heuristic Rewrite Loop Reference

## Purpose

This reference summarizes the planning logic behind the governed KB rewrite loop.
Use it when the rewrite task requires the underlying maintenance model, heuristic taxonomy, or rationale for why rewrite should be selective and governed.

## Core loop

1. Detect
2. Diagnose
3. Draft
4. Review
5. Merge
6. Observe

## Heuristic families

### Length / structure
- very long notes
- weak hierarchy
- oversized bodies without summaries

### Redundancy
- repeated claims across nearby notes
- article/study/context duplication

### Drift
- note type no longer matches original role

### Link mismatch
- linked notes disagree or have become stale relative to one another

### Evidence freshness
- summary or conclusion no longer reflects latest evidence

### Readability
- note is correct but hard to read or hard to share

### Semantic bias / framing risk
- single framing lock-in
- uncertainty collapse
- evidence/claim blur
- option collapse
- premature concept elevation

## Rewrite type families

- summary rewrite
- structure rewrite
- readability rewrite
- team-share rewrite
- evidence-integrating rewrite
- dedupe rewrite
- taxonomy rewrite
- framing rewrite

## Risk framing

### Low
Formatting and low-semantic-impact cleanup.

### Medium
Meaning-preserving but non-trivial restructuring or synthesis.

### High
Conceptual, governance, ontology, or framing changes that may alter interpretation.

## Design principle

The goal is not aggressive rewriting.
The goal is to prevent KB entropy while preserving semantic integrity, traceability, and reviewability.
